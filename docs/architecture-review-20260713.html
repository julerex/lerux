1 · Deepen the fs-server: one server shell, one filesystem-format seam

Strong in-process

userspace/pds/fs-server/src/{leruxfs_handler.rs · fat_handler.rs · block_io.rs}  ·  userspace/crates/{lerux-fs, lerux-fat}

Red bands = same 18 method names, maintained twice. Commit ac58fc5 (multi-client race fix) patched both files: +114 and +116 lines.

After — deep server shell, format adapters

fs-server shell (once)

OpenFile table · job machine · Poll RPC
FsFormat interface
lerux-fs adapter (LERUXFS2)
lerux-fat adapter (FAT16)

Two adapters already exist — the seam is real, it just lives in the wrong place today (cfg-swap of the whole module in main.rs:18–26).

Problem — the seam sits at the whole-handler level: #[cfg(feature = "backend-fat")] swaps 2,464 vs 1,198 lines that duplicate dispatch, open-file table, job machine, and magic client channels; only the format logic actually varies.

Solution — move the seam to a FsFormat interface consumed by one server shell; lerux-fs and lerux-fat become its two adapters.

  • locality: race fixes land once, not twice
  • leverage: one dispatch tested against both adapters
  • delete ~1,000 duplicated lines
  • interface is host-testable (both format crates already are)
Touches ADR-004 positively: async sector I/O currently exists only on the LERUXFS side; a single shell would let FAT inherit it for free.

This crate is the cross-PD compatibility pin — and today it has zero host tests. The interface is the test surface; give it one.

  • delete ~1,400 lines, wire format unchanged
  • locality: length-check bug fixed once
  • leverage: new fields become one-liners
  • wire regressions caught on host, not in QEMU

4 · One source of truth for the board matrix in lerux-cli

Strong local-substitutable

support/boards.toml · tools/lerux-cli/src/{qemu.rs, clippy.rs, test.rs, build.rs}

Before — adding one board touches 6 places

boards.toml — entry (target_triple = target in all 40 rows; serial vars repeated 19×)
qemu.rs:46–327 — ~300-line match from board tag to QEMU args
clippy.rs:24–253 — hand-kept (crate, board-feature) matrices per arch
test.rs — default_curls + needs-disk board-name lists
build.rs:192–205 — test_all board array (33, vs 31 in CI)
smoke-expects.toml — expects (fine: already data)

After — one thick band

boards.toml (the interface)

entry + qemu machine/args or profile key + needs_disk / needs_sp804 / curl flags + in_ci

qemu.rs (derives)
clippy.rs (derives from pds×features)
test.rs (derives)

The consumers become faded internals; the TOML row is the whole interface for “add a board”. Drop the redundant target_triple while at it.

Problem — board knowledge leaks across four Rust modules as hardcoded name lists and match arms; the boards.toml interface is nearly as wide as the implementations it feeds.

Solution — fold QEMU args, lint targets, disk/SP804/curl flags, and CI membership into the board entry (or derive them); delete the parallel matrices.

  • adding a board = one TOML entry
  • delete ~700 lines of match arms and tables
  • test-all vs CI drift becomes impossible
  • qemu.rs / clippy.rs / test.rs become derivations
Aligned with ADR-001’s stated trade-off (“board entries stay numerous until device bindings factor common system_vars sets”) — this is the follow-through, not a contradiction.

The postcard enums are the interface between two PDs, yet each end declares its own copy — the compiler can’t see a mismatch, only QEMU can. The timer pair exists because upstream keeps message_types private (noted in the code comments).

Problem — four private re-declarations of two wire protocols; a one-byte drift becomes a runtime decode failure instead of a compile error.

Solution — move both protocols into shared modules (lerux-interface-types or a tiny lerux-driver-protocols crate); the timer handler body (identical in rdtime/tsc, lines 101–146) can become one generic module over a Clock interface.

  • protocol drift becomes a type error
  • ~120 duplicated lines per timer pair deleted
  • new arch timers implement Clock only

Top recommendation

Deepen the fs-server

It is the largest module in userspace, the duplication is already costing real fixes (the multi-client race patch had to land twice, +114/+116 lines), and both adapters for the FsFormat seam already exist as host-tested crates — the deepening is a reshuffle, not new invention. Candidates 2 and 3 are close seconds and independent enough to run in parallel.
